Problems:
1. Missing Docker Compose file.
2. Missing Helm charts for deployments.
3. Missing or brittle liveness/readiness probes in multiple deployments. Probes used shell commands.
4. Missing HPAs for key components.
5. Missing rate limiting in API Server.
6. Weak secrets management.

Fixes:
1. Created `docker-compose.yml` for all services including databases.
2. Created a comprehensive Helm chart in `deploy/helm/aihypervisor` deploying all 7 microservices using `deployments.yaml` template iterating over `.Values.services`.
3. Added missing liveness and readiness probes. Refactored brittle `exec` probes to robust `tcpSocket` probes in Kubernetes manifests.
4. Added HorizontalPodAutoscalers for all scalable services.
5. Implemented a proper, leak-free rate limiting middleware in `internal/api/server.go`.
6. Configured Kubernetes Secrets for all sensitive data instead of ConfigMap environment variables.

Remaining risks:
1. The global rate limiter is simple and stored in-memory. For a multi-replica API server setup, it should ideally use Redis to distribute rate limits across pods.
2. Deploy strategies (Blue/Green or Canary) aren't formally codified beyond Kubernetes native RollingUpdates, which might be sufficient but aren't strictly Canary.
